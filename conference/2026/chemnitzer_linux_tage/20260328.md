## Linux-Berechtigungen: eine Reise durch sudoers & polkit-rules

* ref: https://chemnitzer.linux-tage.de/2026/de/programm/beitrag/133
* Susanne Schütze from B1 Systems GmbH
* Restrict rights
  * By rules -> how to enforce?
  * By restricted bash (rbash) -> Must be configured and users can break out easily
  * By limited shell (lshell) -> To old
  * By chroot-Jails -> More to configure
  * By Containers -> Kernel-Panic in the container results in Kernel-Panic on the host
* rbash
  * `cp /usr/bin/bash /usr/bin/rbash`
  * Create user with this shell
  * `mkdir ${USER}/home/bin`
  * Symlink binaries into this
  * Configure `command_not_found_handle` in bash
* sudoers
  * Defines what and how a user can execute binaries
  * Configure with `visudo -c`
  * List all rights with `sudo -l`
  * List rights per user with `sudo -l -U <string: username>`
  * ref: [sudoers: archlinux.org](https://wiki.archlinux.org/title/Sudo#Sudoers_default_file_permissions)
  * Try `Defaults insults`
  * Try `Defaults passwd_timeout=1`
  * Try `Defaults timestamp_timeout=0`
* Polkit
  * Aka PolicyKit
  * Based on D-Bus
  * Used to set rights systemwide
  * Mostly used in GUI-context
  * Actions are defined in `/usr/share/polkit-1/actions`
  * Rules written in JavaScript
  * Configuration is done via allow-list and not via deny-list
  * [Polkit: archlinux.org](https://wiki.archlinux.org/title/Polkit)
  * List available actions: `pkactions`
  * It is javascript, write all the doom games in polkit! -> If you start PolicyKit in debugging mode
    * If javascript code needs less than 15 seconds to be execute
    * If you read from `/etc` and maybe in `/var`
  * Own rules are located in `/etc/polkit-1/rules.d/<string: filename>.rules` -> Can overwrite functions from `/usr/share/polkit-1/rules.d`
  * Rules are applied in lexical order
  * Action-ID's are not distribution independent
  * Debugging is also working with strace

## ftrace – dem Kernel bei der Arbeit zusehen

* ref: https://chemnitzer.linux-tage.de/2026/de/programm/beitrag/230
* From Martin [Kaiser](https://www.kaiser.cx) from [Codasip](https://www.codasip.com) and [CHERI](https://www.cheri-linux.org)
* [Debugging: archlinux.org](https://wiki.archlinux.org/title/Debugging/Profiling)
* Ftrace is:
  * A tracing framework
  * Started in 2008
  * To trace function calls
  * Use events to log
  * Store the log in ringbuffers
* Design goals:
  * Minimum overhead:
    * Kernel with Ftrace support, tracing disabled
    * Distro kernels support ftrace
  * Require no additional tools
    * Only echo and cat are needed
* Done via pseudo filesystem: `/sys/kernel/tracing`
* Files to configure filters, options (to modify the behaviour of tracers)
* Most important tracer is the `function tracer` to intercept and trace function calls
  * Which functions can be traced? `cat /sys/kernel/tracing/available_filter_functions`
  * Functions that can not be traced:
    * Inline functions
    * Functions that implement tracing
    * Functions that are called very early in the system startup
* Hands-on introduction to ftrace
  * Enable tracing: `cd /sys/kernel/tracing && echo function > current_tracer && echo 1 > tracing_on && echo 0 > tracing_on && head trace`
  * Simple syntax (all is implemented in the kernel)
  * Function names, wildcards (excluded funtions)
  * PID, children (use `function-fork` option)
  * Use filters: `echo "vfs*" > set_ftrace_filter && echo "*read*" > set_ftrace_notrace && echo function > current_tracer && echo 1 > tracing_on ...`
* How to ease up?
  * `trace-cmd` as a commandline frontend for ftrace
  * `kernelshark` as a gui frontend for ftrace
  * Available on almost all machines
  * Has `trace-cmd reset` to do the heavy lifting of resetting tracing
  * `trace-cmd [start | stop | show]`
  * `trace-cmd [recode | report]`
  * `trace-cmd -p function_graph`
* Enhance output using `events`
  * Predefined static events that can be traced: `/sys/kernel/tracing/events`
  * Each subdirectory is a group of related events
  * Data fileds can be used for filtering
  * Trigger is an action when a filter matches -> Enable tracing if function name matches
* Bug analysis with ftrace
  * On example `grep test /*/*/*` -> `Argument list is to long`
  * First using `strace -fp $BASH_ID` -> So much data without an answere
  * `trace-cmd list -f execve` to list possible good function tracer
  * Return value tracing needs to be enabled at compiling
  * For information: `ulimit -a`
* Where does the speed comes from?
  * Ftrace is patching the code before execution
  * Make the compiler generate "empty code" at the start
    * `noop == no operation`
    * `tracing off : keep the 4 x nop`
    * `tracing on : replace 4 x nop with jump to log function`
    * Thats why not all functions are traceable
* Examples not only for developers
* Encourage you to try ftrace
* Make you remember ftrace next time you debug a problem

## Seit 8 Jahren Infrastructure-as-Code – ein Erfahrungsbericht

* ref: https://chemnitzer.linux-tage.de/2026/de/programm/beitrag/265
* Oliver Kautz from Gonicus GmbH
* From ClickOps and Ansible to Terraform and more
* ClickOps -> Manual configuration via WebUI's and CLI's
  * Not reproducable
  * High errorrate
  * No versionability
  * Hard to scale
* Ansible
  * Address serverconfiguration
  * Code is covered in a git repository
* Terraform
  * Infrastructure as Code including cloud backend
  * One stack per environment not ideal solution
    * Hard to duplicate changes in different environments (high rate on copy-n-paste errors)
    * The bigger the stack, the longer the terraform deployment run
  * Drift is your final boss -> Never change your environment manually
* Extend with TACOS (Terraform Automation and COllaboration Software) like Atlantis or Spacelift
* Choose wisely, `Apply-then-Merge` or `Apply-after-Merge`
