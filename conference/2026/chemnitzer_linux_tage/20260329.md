* `MyClass.__mro__` shows the `method resolution order`, can be shown by `MyClass.mro`
  * `super()` just uses the mro and stopps traveling on first finding
  * [Method Resolution Order: python.org](https://docs.python.org/3/howto/mro.html)

```python
class Type:
    type_name: str

    def print(self) -> None:
        print(f"Type({self.type_name=})")

class Vendor:
    vendor_name: str

    def print(self) -> None:
        print(f"Type({self.vendor_name=})")

class Car(Vendor, Type):
    foo: str

car = Car()
car.type_name = "My Type"
car.vendor_name = "My Vendor"
car.print()

print(f"{Car.__mro__}")
print(f"{Car.mro()}")
```

  * Also possigle `MyClass.function.__doc__`

## Let's Talk About Sex – Programme züchten mit Evolution

* Heiko Spindler
* ref: https://chemnitzer.linux-tage.de/2026/de/programm/beitrag/371
* The goal is to create code that solves a problem by letting the computer create the code
* Benefit of a simulated evolution is, that you just need a small amount of tryouts to find a sweet spot of a solution compared to "try out all possible options"
* A gene is a occurence of this data with the location inside the genome
* Mapping of commands with parameters in a list is the genome
* Goal is to create multiple result sets and rate this results with a fitness score (how well is this evolution adapted to this environment/world)
* Available types:
  * Mutation
  * Inversion
* If you build a fitness function:
  * Add a quality criterion
  * With that you can also negative values to indicate that applying this would result in negativ result (punish)
* First goal is to solve the problem, second goal is to rate the efficiency
* The better results have a higher chance to be taken for the next mutation/iteration
* A speed up would be to start with a "good" generation that has done the first view steps
