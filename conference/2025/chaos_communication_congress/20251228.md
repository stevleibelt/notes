## Über europäische Grenzen hinweg auf klinischen Daten rechnen - aber sicher!

ref: https://events.ccc.de/congress/2025/hub/de/event/detail/uber-europaische-grenzen-hinweg-auf-klinischen-daten-rechnen-aber-sicher

* Das LMU forscht **mit** den Patienten und ihren Daten, nicht **an**
* Das LMU und das LMU Klinikum sind staatliche Einrichtungen
* Selbst entwickelte Software wurde als Open Source [veröffentlicht](https://github.com/lmu)
* Klinische Forschung
  * Findet an (Universitäts-)Kliniken mit Menschen statt
  * Personenbezogene Daten wie Name, Alter, Geschlecht
  * Klinische Daten wie Blutwerte, Bildgebung
* Multizentrische Studien (weltweit durchgeführte Studien)
  * Größere Patientenzahl
  * Besser, wenn es schnell gehen muss
  * Ethnsiche Vielfalt
  * Aber ggf. müssen Daten verarbeitet und getauscht werden
* Komplemetäre Sichtweisen und Konzepte
  * Datenschutz vs. frei zugängliche Daten
  * Mit Kryptografie kann der Bereich für beide verbreitert werden
* Sicheres, verteiltes Rechnen funktioniert durch hinzufügen von "Rauschen"
  * Secure Multiparty Computation (am einfachsten und auch mit kleinen Datenmengen nutzbar)
  * Fully Homomorphic Encryption
  * Differential Privacy
  * Federated Learning
* Secure Multiparty Computation
  * Niemand muss niemanden vertrauen
  * Privacy
  * Security
  * Active Consent (aus passiven Datenlieferanten werden aktive Patienten)
    * Bei jeder neuen Verwendung muss um die Verwendung der Daten mit Begründung/Vorhaben nachgefragt werden
* Prototyp in Deutschland 2019
  * Zwischen der Charite Berlin und LMU München
  * Es funktioniert
  * Langsam, fehlerhaft und ist kompliziert zu entwickeln
* Erste europäische Studie 2024
  * Zwischen Gemelli Rome und LMU
  * [Federated secure computing](https://github.com/federatedsecure)
  * Es funktioniert, ist schnell, leichter zu entwickel und macht keine Fehler
  * Es ist aber noch immer schwer einzusetzen
* **Wichtig**
  * Es hebelt die DSGVO nicht aus
  * Es damit möglich Daten nach Stand der DSGVO möglich
  * Problem: Alle Parteien müssen auf Augenhöhe sein und so auch arbeiten ("Ach komm, gebt mir eure Daten, ich berechne das für euch")
  * Gesellschaftliches Problem: Man muss dem Leadership dazu bringen, dass er aus "Hier habe ich eine Unterschrift unter dem Compliancevertrag, Problem gelöst" herauskriegen
  * [SharedMind](https://sharemind.cyber.ee/) aus Estonia wurde als geschlossene Software eingesetzt

## Auf die Dauer hilft nur Power: Herausforderungen für dezentrale Netzwerke aus Sicht der Soziologie

ref: https://events.ccc.de/congress/2025/hub/de/event/detail/auf-die-dauer-hilft-nur-power-herausforderungen-fur-dezentrale-netzwerke-aus-sicht-der-soziologie

* Wie entstehen und entwickeln sich dezentrale Netzwerke?
  * Autonome Instanzen entwickeln sich unter ähnlichen Rahmenbedinungen
* Welche Herausforderungen müssen diese überwinden
  * Darf nicht über monitären Antrieb verfügen
  * Sie unterliegen einer sozialen Ordnung, die diese nicht einfach zu reparieren ist
  * Es ist einfacher monetäre als gemeinschaftliche Interessen zu organisieren und in Güter oder konkrete Handlungen zu überführen
  * Veringerung des Aufwandens zur Schaffung von Anknüpfungspunkten
  * Man muss die "standardisierte" Weltansicht aufbrechen (Gibt immer nur ein System/Instanz)
  * Selbstverwaltung des Gemeinschaftsguts durch Regeln
* Welche Probleme können dezentrale Netzwerke lösenen?
  * Verringert die Marktabhängigkeiten
  * Widerstandsfähiger gegenüber Sanktionsmechanismen
  * Gesteigerte Souveränität bezüglich eigener Daten
* Merkmale eines dezentralen Netzwerks
  * Es gibt keine zentrale Autorität (Nutzung, Ressourcenkontrolle, Wachstum etc.)
  * Es ist ein Gemeinschaftsgut mit offenen Standards und Protokollen
* Dezentralität in der Praxis
  * Es müssen stets hinreichende Ressourcen mobilisiert werden, um das dezentrale Netzwerk zu ermöglichen
  * Vertrauen (in die Sache) als Vorbedinungen muss gegeben sein
  * Wie hoch ist das Risiko diesen Netzwerk beizutragen (Rechtssicherheit)
* Wie kann man den Problemen entgegenwirken?
  * Alle Teilnehmer tragen einen Teil der Kosten der Infrastruktur bei
  * Verringerung des Aufwands zur Teilhaben an der Infrastrukturdezentralisierung
  * Förderung von Organisationen, Vereine, NGO's etc.
  * Rechtliche Absicherung zentral zu erreichen

## Xous: A Pure-Rust Rethink of the Embedded Operating System

ref: https://events.ccc.de/congress/2025/hub/de/event/detail/xous-a-pure-rust-rethink-of-the-embedded-operating-system

* Pure ideas
  * No POSIX
  * Memory safty
* MMU's are available in Chips sind 1985
* But cheap chips don't have a MMU (e.g. ARM7TDMI)
* So cheap os where created that don't use a MMU
* Powercycle established, no MMU needed, no MMU produced
* But Linux needs a MMU, so ARM creates a `Cortex-A` as high priced cpu while keeping `Cortex-M` as cheap
* RISC-V has a configureable MMU
* So they've created a SoC with an MMU
* So they've started creating Xous
* What kind of MMU is needed?
  * Don't need entire address space
  * Don't need page-sized granularity
  * Don't need multi-level page tables
  * Need process isolation
  * Need fast switching
* Which results into totally isolated processes, kernel included
* No process can talk to each other, so they need message passing
* What is needed?
  * Send scalars
  * Send references/memory -> detaching memory from one process and linking to another is doable in rust
* Inspired by:
  * HelenOS
  * QNX
  * Solaris (Doors?)
* Ticktimer
  * Takes ownership of hardware timer
  * Handles uptime count
  * Provides scheduler
    * ReturnToParent syscall
  * Provides msleep
* Results
  * Upstream Tier-3 Rust Target
  * Only using stable features
  * No compiler or linker needed
  * Rust semantics enforced by hardware MMU
* Problem, building chips is expensive
* Fully [open source](https://baochip.com/) chip available now
